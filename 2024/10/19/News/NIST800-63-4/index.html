# Prologue

近期實習的資安小文章都在做密碼管理，先是密碼管理器，再是無密碼機制 FIDO，交付的時候 Elisa 剛好看到一篇有關 PCHOME 疑似遭到撞庫攻擊的新聞，就問我要不要順便補個時事上 FIDO 那篇（不過目前也沒確定 PCHOME 是不是真的遭到撞庫攻擊，所以沒有真的寫上去，相關資訊可以看這篇說明），但我的認知中撞庫攻擊是因為使用者懶的記憶太多組密碼所以只拿同一組密碼打天下，才讓壞人有機會到處撞來撞去就入侵一堆該用戶名下的帳戶，又很剛好的 NIST 在八月底九月初提出了新一版的數位身分指南 (Digital Identity Guidelines)，就問 Elisa 要不要順便把這篇整理起來當作下一篇資安小文章，於是就誕生了這篇新知分享，畢竟資安小文章對於字數、篇幅都有限制，在這邊可以恣意的撰寫 XD

這篇比較會用參照本文結合一些我自己的想法來記錄，不想看那麼多的可以滑到最下面看重點總整理，或是直接看中文的部分

NIST SP 800-63 Digital Identity Guidelines 並不是單純針對「什麼是安全密碼」進行規範，先來看看本文怎麼說

These guidelines cover identity proofing and authentication of users (such as employees, contractors, or private individuals) interacting with government information systems over networks. They define technical requirements in each of the areas of identity proofing, registration, authenticators, management processes, authentication protocols, federation, and related assertions. They also offer technical recommendations and other informative text intended as helpful suggestions. The guidelines are not intended to constrain the development or use of standards outside of this purpose. This publication supersedes NIST Special Publication (SP) 800-63-3.

也就是說這份指南包含對用戶的身分證明、身分驗證進行規範，而用戶的定義包括使用網路與政府資訊系統進行互動的員工、承包商、個人。規範內容包含身分證明、註冊、驗證、管理等等，除了規範外，這份指南也有提供一些技術上的建議，不過對於與政府互動以外的地方，並沒有限制用戶一定要完全遵照這份指南

雖然 NIST 的這份指南涵蓋範圍很廣，但這篇新知分享會著重在「什麼是安全的密碼」

# Authentication Assurance Levels (AAL)

進到正題之前，我覺得有必要討論一下在 NIST SP 800-63B 提到的身分驗證保證等級，

# AAL1

AAL1 provides basic confidence that the claimant controls an authenticator bound to the subscriber account being authenticated. AAL1 requires only single-factor authentication using a wide range of available authentication technologies. However, it is recommended that applications assessed at AAL1 offer multi-factor authentication options. Successful authentication requires that the claimant prove possession and control of the authenticator.

第一級也是對機密性要求最基本的層級，驗證方式只要單一驗證即可，但 NIST 仍推薦使用多因子驗證，主要驗證因子在驗證其擁有性，允許驗證器類別包含下列幾種：

  • Password：使用者自行選擇並記憶的 token
  • Look-up secret：從使用者持有的清單中尋找提示值決定
  • Out-of-band device：透過一條特殊通道來進行驗證
  • Single factor OTP：透過使用者持有的設備進行一次性密碼驗證
  • Multi-factor OTP：透過使用者持有的設備進行一次性密碼驗證，但在此之前須經過其他驗證因子 (2FA, MFA)
  • Single-factor cryptographic authentication：透過加密金鑰進行驗證
  • Multi-factor cryptographic authentication：透過加密金鑰進行驗證，但在取得金鑰前須經過其他驗證因子 (2FA, MFA)

# AAL2

AAL2 provides high confidence that the claimant controls one or more authenticators bound to the subscriber account being authenticated. Proof of the possession and control of two distinct authentication factors is required. Applications assessed at AAL2 must offer a phishing-resistant authentication option.

第二級的機密性較高，驗證方式需要多重驗證，且必須防止網路釣魚，驗證因子則是在驗證擁有性與是否具有兩種不同的身分驗證因子。允許驗證器類別如下

  • Multi-factor authenticator

    可以再添加下列機制

    • Multi-factor Out-of-band authenticator
    • Multi-factor OTP
    • Multi-factor cryptographic authentication
  • Combination of two single-factor authenticators

    Combination 還要 password 和一個實體驗證器 (something you have)

    • Look-up secret
    • Out-of-band device
    • Single-factor OTP
    • Single-factor cryptographic authentication

# AAL3

AAL3 provides very high confidence that the claimant controls one or more authenticators bound to the subscriber account being authenticated. Authentication at AAL3 is based on the proof of possession of a key through the use of a public-key cryptographic protocol. AAL3 authentication requires a hardware-based authenticator with a non-exportable private key and a phishing-resistant authenticator (see Sec. 3.2.5); the same device may fulfill both requirements. To authenticate at AAL3, claimants are required to prove possession and control of two distinct authentication factors.

第三級機密性極高，驗證方式要求透過公開金鑰加密協定證明擁有金鑰、基於硬體的身分驗證器、不可匯出私鑰，且仍需防止網路釣魚，驗證因子仍為證明其擁有性與具備兩種不同身分驗證因子。允許驗證器類別為以確認安全的密碼系統

# Authenticator and Verifier Requirements

在 Section 3 的部分，NIST 開始說明到對於驗證器的要求，不過原文中會以這些詞彙區分規範嚴謹程度，程度由高至低如下：

💡 要求階級定義

要求階級定義 說明
SHALL / SHALL NOT 嚴格遵守，不允許偏離
SHOULD / SHOULD NOT 幾種可能性中被推薦的
MAY / NEED NOT 不鼓勵，不禁止，只要按照規則上說的走就好
CAN / CAN NOT 因為一些實質或邏輯上的原因所以可以或不行這樣做

# 3.1 Requirements by Authenticator Type

# 3.1.1 Password

A password (sometimes referred to as a passphrase or, if numeric, a PIN) is a secret value intended to be chosen and either memorized or recorded by the subscriber. Passwords must be of sufficient complexity and secrecy that it would be impractical for an attacker to guess or otherwise discover the correct secret value. A password is “something you know”.

NIST 在這邊說到 password 表示「你知道」，而不是「你是誰」

Passwords SHALL either be chosen by the subscriber or assigned randomly by the CSP.

密碼應該是使用者自己選擇的或是 CSP 隨機產生的

  • 使用者自己選擇的：他應該要知道
  • CSP 隨機產生的：CSP 應該要知道

If the CSP disallows a chosen password because it is on a blocklist of commonly used, expected, or compromised values (see Sec. 3.1.1.2), the subscriber SHALL be required to choose a different password. Other complexity requirements for passwords SHALL NOT be imposed. A rationale for this is presented in Appendix A, Strength of Passwords.

CSP 可以因為用戶使用常見的或容易猜測的密碼而拒絕該組密碼，此時用戶應該重新提交一組密碼，但 CPS 不能基於複雜度要求施加其他限制，例如：請輸入包含英文大小寫、數字、特殊符號 blablabla

3.1.1.2 Password Verifiers

Verifiers and CSPs SHALL require passwords to be a minimum of eight characters in length and SHOULD require passwords to be a minimum of 15 characters in length.

密碼長度至少要 8 個字元，建議超過 15 個字元

Verifiers and CSPs SHOULD permit a maximum password length of at least 64 characters.

系統可支援密碼長度至少要超過 64 個字元

Verifiers and CSPs SHOULD accept all printing ASCII characters and the space character in passwords.

應該要接受所有 ASCII 可視字元

Verifiers and CSPs SHOULD accept Unicode characters in passwords. Each Unicode code point SHALL be counted as a single character when evaluating password length.

建議要支援所有 Unicode，每個 Unicode code point 視為一個字元

Verifiers and CSPs SHALL NOT impose other composition rules (e.g., requiring mixtures of different character types) for passwords.

不能對密碼施加其他複雜度要求，比如說「應該同時包含特殊符號、大寫英文字母、小寫英文字母，不符合要求就拒絕你的密碼申請 ಠ౪ಠ」

Verifiers and CSPs SHALL NOT require users to change passwords periodically. However, verifiers SHALL force a change if there is evidence of compromise of the authenticator.

不能要求用戶定期更換密碼，除非有證據說這組密碼已外洩

Verifiers and CSPs SHALL NOT permit the subscriber to store a hint that is accessible to an unauthenticated claimant.

對未驗證之用戶不應該提供密碼提示，就是說不能提供「不知道密碼嗎？那你告訴我你家養的寵物是什麼？知道我就當作你通過」

Verifiers and CSPs SHALL NOT prompt subscribers to use knowledge-based authentication (KBA) (e.g., “What was the name of your first pet?”) or security questions when choosing passwords.

用戶註冊密碼時，也不應該詢問知識型驗證問題 (KBA)，簡單來說就是關於 KBA 都不要用

Verifiers SHALL verify the entire submitted password (i.e., not truncate it).

驗證密碼時應該針對整組輸入密碼進行驗證，而非片段，反例如下：

When processing a request to establish or change a password, verifiers SHALL compare the prospective secret against a blocklist that contains known commonly used, expected, or compromised passwords. The entire password SHALL be subject to comparison, not substrings or words that might be contained therein. For example, the list MAY include but is not limited to:

  • Passwords obtained from previous breach corpuses
  • Dictionary words
  • Context-specific words, such as the name of the service, the username, and derivatives thereof
    If the chosen password is found on the blocklist, the CSP or verifier SHALL require the subscriber to select a different secret and SHALL provide the reason for rejection. Since the blocklist is used to defend against brute-force attacks and unsuccessful attempts are rate-limited, as described below, the blocklist SHOULD be of sufficient size to prevent subscribers from choosing passwords that attackers are likely to guess before reaching the attempt limit.
    Excessively large blocklists are of little incremental security benefit because the blocklist is used to defend against online attacks, which are already limited by the throttling requirements described in Sec. 3.2.2.

用戶創建或更新密碼時，系統應該應該驗證整段 password，而不是其中的子字串，且系統內 blocklist 上的皆應該要驗證到，Blocklist 可能包含之前洩漏出去的密碼、特定字典上的字、特定字詞，例如系統名稱、用戶名或其他可推導之字詞，如果有違反上述情形，要拒絕用戶的註冊，並提供原因。而 blocklist 的 size 應該足夠大，衡量準則是攻擊者無法在嘗試次數內輸入到正確的用戶密碼，但 blocklist 的 size 不是越大越好，太大對安全性的增加並沒有太大的影響

Verifiers SHALL offer guidance to the subscriber to assist the user in choosing a strong password.

系統應該提供用戶安全密碼準則，讓用戶有機會可以知道如何設定才是安全的密碼

Verifiers SHALL implement a rate-limiting mechanism that effectively limits the number of failed authentication attempts that can be made on the subscriber account

系統要提供嘗試次數限制，顯然是為了增加攻擊者攻擊成本

Verifiers SHALL allow the use of password managers. Verifiers SHOULD permit claimants to use the “paste” functionality when entering a password to facilitate their use. Password managers have been shown to increase the likelihood that users will choose stronger passwords, particularly if the password managers include password generators.

這邊提到系統應該允許密碼管理器、使用貼上輸入密碼，從這裡可以看到 NIST 認為密碼管理器是可信任的，NIST 說使用密碼管理器的用戶比較有可能使用安全的強密碼，尤其是密碼管理器支援自動產生密碼的功能，畢竟不需要自己想也不用記，大家就有餘力追求安全，這在這一版的指南是很重要的觀念，NIST 開始強調「當準則不讓使用者感到負擔，用戶才願意主動追求安全」

To assist the claimant in successfully entering a password, the verifier SHOULD offer an option to display the secret — rather than a series of dots or asterisks (*) — while it is entered and until it is submitted to the verifier. This allows the claimant to confirm their entry if they are in a location where their screen is unlikely to be observed. The verifier MAY also permit the claimant’s device to display individual entered characters for a short time after each character is typed to verify the correct entry. This is common on mobile devices.

系統可以用一堆點點或星號擋住用戶輸入的密碼，但至少要提供一個眼睛按鈕或其他方法讓用戶有方法可以看到自己輸入了什麼，畢竟 typo 是無可避免的，像是 Linux 打了 sudo 指令，系統跟用戶討密碼時，用戶輸入了什麼都看不到，這就不符合 NIST 這邊說的。而 NIST 還提到在行動裝置上，可以讓剛輸入的密碼顯示幾秒鐘再變成點點或星號，這樣對用戶也比較容易觀察自己輸入了什麼

Verifiers MAY make allowances for mistyping, such as removing leading and trailing whitespace characters before verification or allowing the verification of passwords with differing cases for the leading character, provided that passwords remain at least the required minimum length after such processing.

系統可以新增一個 feature 是 truncate 掉開頭跟結尾的空白，只要符合前述密碼規定 (e.g., length)，這也是因為避免用戶 typo 的設計，但可以注意的是 NIST 用 MAY 表示這個強制力很低，可以不用

Verifiers and CSPs SHALL use approved encryption and an authenticated protected channel when requesting passwords.

跟用戶討密碼時，應該都是經過加密的通道傳輸，不然就洩漏了

Verifiers SHALL store passwords in a form that is resistant to offline attacks. Passwords SHALL be salted and hashed using a suitable password hashing scheme.
系統要用加鹽雜湊儲存密碼，這是為了防範 offline attacker
Password hashing schemes take a password, a salt, and a cost factor as inputs and generate a password hash. Their purpose is to make each password guess more expensive for an attacker who has obtained a hashed password file, thereby making the cost of a guessing attack high or prohibitive.
hash 機制的輸入包含 password, salt 和 cost factor。Cost factor 可以想成 round 數，cost factor 大，表示運算成本越高。目的是比起一般 hash 增加攻擊難度，畢竟如果攻擊者知道 hash algorithm 和一些 key 這個 hash 就沒用了
The chosen cost factor SHOULD be as high as practical without negatively impacting verifier performance. It SHOULD be increased over time to account for increases in computing performance. An approved password hashing scheme published in the latest revision of [SP800-132] or updated NIST guidelines on password hashing schemes SHOULD be used. The chosen output length of the password verifier, excluding the salt and versioning information, SHOULD be the same as the length of the underlying password hashing scheme output.
- Cost factor 越大越好，但不能對 verifier 的驗證效率產生負面影響
- hash 應該要用之前準則提出來的，也就是說不能用危險的 hash，例如 SHA1
- 至於密碼長度合理與否的判斷是在不含鹽的情況下審視的

The salt SHALL be at least 32 bits in length and chosen to minimize salt value collisions among stored hashes. Both the salt value and the resulting hash SHALL be stored for each password. A reference to the password hashing scheme used, including the work factor, SHOULD be stored for each password to allow migration to new algorithms and work factors. For example, for the Password-Based Key Derivation Function 2 (PBKDF2) [SP800-132], the cost factor is an iteration count: the more times that the PBKDF2 function is iterated, the longer it takes to compute the password hash.

鹽的長度至少要 32-bit，且要避免鹽之間也發生 collision 的狀況。每組 password 應該都會對到一個 salt 和一個 hash result，這是為了因應當更換 hash 方法或 cost factor 的時候

In addition, verifiers SHOULD perform an additional iteration of a keyed hashing or encryption operation using a secret key known only to the verifier. If used, this key value SHALL be generated by an approved random bit generator, as described in Sec. 3.2.12. The secret key value SHALL be stored separately from the hashed passwords. It SHOULD be stored and used within a hardware-protected area, such as a hardware security module or trusted execution environment (TEE). With this additional iteration, brute-force attacks on the hashed passwords are impractical as long as the secret key value remains secret.

NIST 建議 verifier 應該要多提供一層 key hashing 或是加密，當然如果有做這一步的話，key 的值要是一個驗證過確定夠安全的 random bit generator 產生，且要放在硬體保護安全的地方

# 重點總整理

  1. 密碼長度遠比複雜度重要
  2. 不要在不同平台重複使用密碼，避免遭到撞庫攻擊
  3. 不再使用提示等知識型驗證機制
  4. 不再定期更換密碼，除非密碼遭洩漏
  5. 推薦使用合規的密碼管理器
  6. 啟用多因子驗證，加強安全性

雖然說這份數位身分驗證指南仍只是草案，但可以看到 NIST 在這個版本中對於讓使用者方便記憶管理密碼這件事情下了一些功夫，因為太過複雜的方式反而會讓使用者對於安全這件事卻步，甚至因為對安全無感而使用更不安全的方式。
